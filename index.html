Do Agents Handle Code Clones Like Humans? An Exploratory Study on GitHub Pull Requests

Abstract

Context: Large Language Models (LLMs) and autonomous coding agents are increasingly involved in software maintenance tasks, including code refactoring. Among these tasks, code clone removal is suited for automation. However, little is known about how AI agents actually perform clone refactoring in real-world development workflows compared to human developers. Objectives: The goal of this paper is to investigate how AI agents per- form code clone refactoring through pull requests. We aim to compare the efficiency and review dynamics of clone-related PRs authored by pure humans, human–created but agent-generated, and fully agents. We also aim to under- stand the qualitative differences in the way humans and agents describe, justify, and revise their clone refactorings. Methods: This paper presents an empirical study of 300 clone-related pull requests (PRs) from the AIDev dataset, equally divided into three groups: (1) PRs created and written by humans, (2) PRs created by humans but generated by agents, and (3) PRs created and written entirely by agents. Results: Our quantitative analysis shows that agent-involved PRs are com- pleted significantly faster, with smaller change sizes and lower reviewer work- load compared to pure human PRs (δ = 0.40–0.59, p < 0.001). Qualitative analysis reveals that 42.2% of human-created but agent-generated PRs show a comprehensive structure and explicit definition of scope, improving readability and review efficiency. In contrast, pure human PRs show more proactive self- correction, though at higher cost in time and review effort. A small proportion of clones remained unresolved across all PR types (6% in human PRs, 1% in human-created but agent-generated PRs, and 5% in agent PRs). Among all PRs, none explicitly mentioned clone types such as Type-1 or Type-2 in title, description, or inline comments. For refactoring strategies, extraction-based refactoring was the most common approach (56%–61%), followed by direct re- moval (34%–44%), while pull-up refactoring appeared only in agent PRs. Conclusion: Overall, our findings show that AI agents are effective for clone refactoring tasks requiring precision and speed, but they still rely on human oversight when refactoring involves multiple files and requires understanding the broader project structure.

Python Scripts

The scripts are inside the Scripts/ folder. And here is our the GitHub repository Requirements.txt includes the necessary Python packages. It is in the Scripts folder as well. Install: pip install -r requirements.txt

  • get_dataset.py — Download a table from the AIDev dataset and save it as CSV and a SQLite DB.
    Run: python get_dataset.py (edit the table variable inside the file to choose the table).
  • get_clone_data.py — Find clone‑related pull requests in a CSV using keyword matching and write matches to review_clone.csv in the output folder.
    Run: python get_clone_data.py --file all_pull_request.csv --out_dir out/
  • get_code_human.py — Download code changes for human‑written PRs from GitHub (commits, files, diffs).
    Run: python get_code_human.py
  • get_refactor.py — Find refactor‑tagged PRs from pull_request.csv and save subsets as CSV/DB files under refactor_data/curated_agent/.
    Run: python get_refactor.py
  • pr_code_scraper.py — Helper to fetch code artifacts for PRs via the GitHub API (used by other scripts or standalone).
    Run: python pr_code_scraper.py or import it in Python.
  • merge_sqlite_dbs.py — Merge many SQLite DBs in a folder into a single database.
    Run: set source_folder and output_database at the top of the file, then run python merge_sqlite_dbs.py.
  • run_rminer.py — Generate or execute RefactoringMiner commands for selected PR IDs.
    Run (print commands): python run_rminer.py --csv human_pull_request.csv --ids 2625652637,2474497456 --out /tmp/rminer.tsv
    Run (execute): python run_rminer.py --csv human_pull_request.csv --ids-file pr_ids.txt --rminer /path/to/RefactoringMiner --results-dir rminer_results
  • get_pr_metrics.py — Calculate quantitative metrics for pull requests, including size, time to close, comments, and churn, from raw PR data. Outputs full metrics and summary metrics CSV files.
    Run: python get_pr_metrics.py --csv pull_request.csv --out_full metrics_full.csv --out_summary metrics_summary.csv
    You can run this script multiple times with different input CSVs or output file names to generate separate metrics files for each group (e.g., curated_human_created_metrics.csv and pure_human_metrics.csv).
  • compare_pr_metrics.py — Compare PR metrics between human, hybrid, and agent groups using Mann–Whitney U tests and effect size calculations. Generates comparison tables, summary CSVs, and plots for statistical analysis.
    First, run get_pr_metrics.py twice (once for each PR group) to generate two metrics files. Then use those files as inputs:
    Run: python compare_pr_metrics.py --file-a curated_human_created_metrics.csv --file-b pure_human_metrics.csv --label-a Curated_pr --label-b Pure_human
  • get_number_reviewer.py - Calculate the number of unique reviewers for each PR and add it as a metric to the metrics CSV files.
    Run: Get the github api key and set the path to the csv files that have PR urls

Experimental Data

The experimental data used in this study are derived from the AIDev dataset and its related GitHub mining results. The data can be downloaded here. Below are brief descriptions of each dataset:

  • all_pull_requests — A large master dataset containing metadata for all pull requests across repositories, including IDs, descriptions, timestamps, and repository links.
  • all_repositories — Contains metadata about all repositories in the dataset, including repository ID, name, owner, and project language.
  • all_user — Includes detailed user information such as user ID, username, and account type (human or agent).
  • human_pull_request — The main dataset of pull requests created by humans. Each record includes PR ID, title, description, timestamps, and repository linkage.
  • human_pr_task_type — Provides task classification for each human PR, such as bug fix, refactor, or feature addition.
  • issue — Lists issues linked to PRs, including issue titles, descriptions, and repository references.
  • pr_comments — Contains textual comments left by contributors and reviewers within PR discussions.
  • pr_commits — Includes commit-level details for each PR, such as commit IDs, timestamps, and commit messages.
  • pr_review_comments — Contains review comments made for curated PRs during code review.
  • pr_reviews — Contains general review metadata for each pull request, including reviewer IDs, states, and timestamps.
  • pr_task_type — Lists task categories of pull requests, such as feature, bug fix, or refactor.
  • pr_timeline.csv — Records key timeline events for each pull request, including creation, first review, and merge times.
  • curated_pull_request — Includes all pull request metadata across the dataset, such as ID, title, author, and status.
  • related_issue — Maps curated pull requests to their related issues, linking their respective IDs.
  • repository — Contains detailed curated repository-level information including names, languages, and creation times.
  • user — Provides user information for curated PRs such as user IDs, usernames, and account attributes.
  • mined_code - Includes the code artifacts (commits, files, diffs) for related human PRs and curated PRs.
  • clone_related_data - Includes the clone-related PRs for title, body and both identified from the AIDev dataset using keyword matching.

Results

Quantitative Analysis Results

Metric Definitions

We report the following quantitative metrics for each curated group. All metrics are computed from the AIDev pull request data and GitHub mining results.

  • time_to_close_hours — Time between PR creation and closing (merged or closed) measured in hours.
  • time_to_close_days — The same closing time interval expressed in days.
  • body_length — Length of the PR description body, measured as the number of characters in the text.
  • commits — Total number of commits associated with the pull request.
  • changed_files — Number of files that were changed (added, modified, or deleted) in the PR.
  • additions — Total number of added lines across all changed files in the PR.
  • deletions — Total number of removed lines across all changed files in the PR.
  • code_churn — Sum of additions and deletions, used for overall change size.
  • review_iterations — Number of distinct review rounds for the PR, approximated from the sequence of review events and states.
  • total_comments — Total count of review comments and general PR comments linked to the pull request.
  • reviewer_workload_hours — Approximate reviewer response time, measured as the average number of hours between when reviewers are requested to review a PR and when they submit their reviews.
  • Number_of_Reviewers — The count of unique reviewers who participated in the review process for the PR.
